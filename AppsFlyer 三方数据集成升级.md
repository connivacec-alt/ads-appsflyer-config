# AppsFlyer 接入配置升级需求文档

## 1. 背景
当前三方数据接入配置页面中，AppsFlyer 接入能力存在以下限制：
1. **平台写死**
   - 页面当前仅展示 `ANDROID`、`IOS`
   - 后续若新增 `WINDOWS` 等平台，需要改造页面结构
2. **AppID 只能单个配置**
   - ANDROID 仅支持 1 个 AppID
   - IOS 仅支持 1 个 AppID
   - 新项目测试阶段常出现同平台多个包并行测试的场景，现有配置无法满足
3. **事件采集规则写死**
   - 事件采集规则中，当前固定使用 `af_complete_registration`
   - 但不同项目在 AppsFlyer 后台配置的事件名称并不一致
   - 需要支持可配置化
---

## 2. 目标
本次改造目标为：
1. 支持**平台可扩展**
2. 支持**同一平台下多个 AppID 配置**
3. 支持**事件名按平台维度配置**
4. 支持**启用/停用状态**
5. 保持现有接入逻辑兼容，降低改造风险
---

## 3. 需求范围
### 3.1 平台配置
支持平台维度的可扩展配置，不再仅限于 ANDROID、IOS。
### 3.2 AppID 配置
支持每个平台配置多个 AppID。
### 3.3 事件规则配置
支持按平台配置该平台在 AppsFlyer 中实际使用的上报事件名，不再写死 `af_complete_registration`。
### 3.4 状态配置
每个平台配置项支持：
- 启用
- 停用
---
## 4. 页面方案
### 4.1 页面结构
页面采用垂直布局，包含以下模块：
1. **顶部信息栏** - AppsFlyer Logo、平台名称、接入状态标签
2. **APPID及注册事件配置** - 平台配置卡片（支持多平台、多AppID）
3. **Push API 数据接收配置** - 启用开关、地址复制、定时更新说明
4. **Pull API 数据接收配置** - 启用开关、API Token输入、定时更新说明
5. **底部操作栏** - 确定/返回按钮（固定吸底）

---
### 4.2 APPID及注册事件配置
每个平台配置项包含：
- 平台标签（ANDROID/IOS/HARMONYS/WEB/WECHATGAME/WINDOWS）
- AppID 列表（标签形式，支持回车添加和删除）
- 注册事件输入框
- 删除操作（带二次确认）

**新增平台**：通过下拉菜单选择添加

**平台列表展示示例**：
| 平台 | AppID | 注册事件 | 操作 |
|---|---|---|---|
| ANDROID | com.xxx.app, com.xxx.beta | af_complete_registration | 删除 |
| IOS | id1463021419 | ios_complete_registration | 删除 |
| HARMONYS | com.xxx.harmony | harmony_register | 删除 |

---
### 4.3 Push API 数据接收配置
- **禁用状态**：显示步骤引导（①开启开关 ②复制地址 ③前往AppsFlyer配置）和提示说明
- **启用状态**：显示数据接收地址（可复制）和定时更新标签
- 状态标签颜色：浅黄底+橙色图标+深灰文字
- 删除确认气泡框：左侧弹出，红色确认按钮

---
### 4.4 Pull API 数据接收配置
- **禁用状态**：显示步骤引导和提示说明
- **启用状态**：显示API Token输入框和定时更新标签
- 提示文字：浅灰色，链接指向AppsFlyer帮助文档

---
### 4.5 状态标签设计
| 状态 | 样式 |
|---|---|
| 待配置 | 浅黄底(#fffbe6) + 橙色边框(#ffe58f) + 深灰文字 + 沙漏图标 |
| 已关闭 | 浅灰底(#fafafa) + 灰色文字 |
| 已接入 | 浅绿底(#f6ffed) + 绿色文字 + 对勾图标 |
| 警告 | 浅红底(#fff1f0) + 红色边框(#ffccc7) + 深灰文字 + 感叹号图标 |

---
### 4.6 交互细节
- **删除操作**：点击后弹出气泡确认框，二次确认后执行删除
- **步骤引导**：数字使用圆形背景（#d9d9d9），箭头使用 "›" 符号
- **复制按钮**：白底蓝边框，悬浮时填充蓝色背景
- **固定吸底**：底部操作栏固定在页面底部，背景毛玻璃效果
- **输入框**：注册事件宽度200px，AppID输入框最小宽度240px

---
## 5. 交互说明
### 5.1 新增平台
点击 `+ 添加平台配置` 后：
1. 弹出平台选择框
2. 选择平台类型
3. 系统生成一条配置项

### 5.2 AppID 配置
- 每个平台支持多个 AppID
- AppID 支持手动添加
- 支持删除某个 AppID
- 同一平台内 AppID 不可重复

### 5.3 事件名配置
- 每个平台可配置对应的 AppsFlyer 事件名
- 默认为 `af_complete_registration`
- 支持编辑修改

### 5.4 状态配置
- 每个平台配置项支持启用/停用
- 停用后该平台配置不参与后续匹配逻辑
- 默认状态建议为启用

---

## 6. 校验规则

### 6.1 平台校验
- 平台不能为空
- 平台不能重复配置

### 6.2 AppID 校验
- AppID 不能为空
- 同一平台下 AppID 不能重复
- AppID 格式需符合平台要求
  - ANDROID：包名格式
  - IOS：App Store ID 或系统约定格式
  - WINDOWS：按业务约定格式

### 6.3 事件名校验
- 事件名不能为空
- 需符合 AppsFlyer 后台实际配置名称
- 建议支持英文、数字、下划线

### 6.4 状态校验
- 状态只能为：启用 / 停用

---

## 7. 数据结构建议

建议后端采用数组结构存储平台配置，避免写死平台字段。

```json
{
  "platform_configs": [
    {
      "platform_code": "ANDROID",
      "platform_name": "ANDROID",
      "app_ids": ["com.xxx.app", "com.xxx.beta"],
      "event_name": "af_complete_registration",
      "status": 1
    },
    {
      "platform_code": "IOS",
      "platform_name": "IOS",
      "app_ids": ["id1463021419"],
      "event_name": "IOS_complete_registration",
      "status": 0
    }
  ]
}
```

字段说明：
- `platform_code`：平台编码
- `platform_name`：平台名称
- `app_ids`：该平台下的 AppID 列表
- `event_name`：该平台对应的 AppsFlyer 事件名
- `status`：状态，`1` 表示启用，`0` 表示停用

---

## 8. 兼容性说明

### 8.1 现网配置兼容
需兼容当前已有的单 AppID 配置方式：
- ANDROID 单值配置
- IOS 单值配置

建议支持旧数据自动迁移到新结构。

### 8.2 接口兼容
如现有接口入参为单值字段，建议新增数组字段，同时保留旧字段一段时间，便于平滑迁移。

---

## 9. 研发实现建议

### 9.1 前端
- 平台列表动态渲染
- AppID 使用 Tag/多值输入组件
- 事件名使用普通输入框
- 状态使用开关组件
- 支持新增、删除、编辑

### 9.2 后端
- 配置结构改为平台数组
- 数据校验支持平台唯一性、AppID 唯一性
- 回调事件匹配时，按平台 + AppID + 事件名 + 状态进行判断

### 9.3 数据匹配逻辑
当 AppsFlyer 回调数据进入系统后：
1. 根据平台判断所属配置
2. 根据 AppID 在配置列表中匹配
3. 根据事件名匹配当前平台配置的采集事件
4. 仅当配置状态为启用时，执行后续归因逻辑

---

## 10. 优先级建议

### P0
- 支持多 AppID 配置
- 支持事件名按平台维度配置
- 支持启用/停用状态

### P1
- 支持平台动态扩展
- 支持新增 WINDOWS 平台

---

## 11. 验收标准

1. ANDROID 可配置多个 AppID
2. IOS 可配置多个 AppID
3. 平台配置不再写死，支持后续新增平台
4. 事件采集规则支持按平台配置自定义事件名
5. 每个平台配置支持启用/停用
6. 现有配置可正常迁移和使用
7. 配置保存后，回调匹配逻辑正常生效
